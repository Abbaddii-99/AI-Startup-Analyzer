# نظرة سريعة مبسطة

## ١. لمحة عامة تخلّيك تفهم الهيكل في ثانية
- **أيقونة المشروع**: `apps/` للتطبيقات التنفيذية، `packages/` للكتب المشتركة، وملفات الجذر للضبط (pnpm، turbo، وثائق، وملفات البيئة).
- **الهدف الأساسي**: واجهة مستخدم Next.js تُرسل أفكارًا إلى Backend NestJS يتم فيها استدعاء وكلاء ذكاء اصطناعي، ثم تُعرض تقارير تحليلية شاملة.
- **أهمية الملف**: هذه الوثيقة توضح باختصار كل طبقة، تميّز الحساسيات، وتقترح خطوات قبل تحويل Repos إلى خاص.

## ٣. ملفات ومكونات رئيسية (لما تفتح الملف تعرف وش يسوي بسرعة)
| المسار | الدور المحوري |
|---|---|
| `apps/backend/src/app.module.ts` | يربط Config/Bull/Throttler والـ Modules (Auth, Analysis, Agents). يعرف الـ ThrottlerGuard الشامل. |
| `apps/backend/src/main.ts` | يفعِّل CORS، الـ ValidationPipe، وفلتر الأخطاء، ويشغّل الخادم على `process.env.BACKEND_PORT`. |
| `apps/backend/src/analysis/analysis.service.ts` | ينشئ تحليلًا جديدًا، يطبّق قيود الخطة، يستدعي وكلاء متعددين عبر الطابور ويعيد البيانات بعد انتظار التنفيذ. |
| `apps/backend/src/queue/analysis.processor.ts` | ينسق معالجة الـ job، يدير التقدم، يُشغّل الوكلاء في تسلسل، يطبّق معالجات grounding، ويحدّث سجل `analysis` النهائي في Prisma. |
| `apps/backend/src/agents/ai.service.ts` | موصل صالح لـ Gemini أو OpenRouter، يدير Redis للكاش، retries، ويستخرج JSON المستجيب من الـ Prompt. |
| `apps/backend/src/auth/*` | تسجيل/تسجيل دخول/refresh/logout باستخدام `AuthService` و`JwtService`، مع دعم Google OAuth (cookies + redirect). |
| `apps/frontend/src/app/page.tsx` | واجهة البداية، نموذج إدخال الفكرة مع قيود `MAX_IDEA_LENGTH`، يطلب تسجيل دخول قبل استدعاء `/analysis`. |
| `apps/frontend/src/app/analysis/[id]/page.tsx` | صفحة تقرير مفصل (progress polling، scoreboard، إعادة توليد أقسام، دردشة + تنزيل PDF). |
| `apps/frontend/src/components/Sidebar.tsx` | قائمة جانبية ثابتة بها تنقل الأقسام، زر "Ask AI"، وعرض بيانات المستخدم وخروج مع `api.post('/auth/logout')`. |
| `apps/frontend/src/lib/api.ts` | axios مع Base URL من `NEXT_PUBLIC_API_URL`، إضافة Authorization header من zustand/localStorage. |
| `apps/frontend/src/lib/store.ts` | zustand + persist لحفظ `token`, `refreshToken`, `user`, مع دوال `logout`, `setAuth`. |
| `packages/shared/src` | مكتبة توحيد (types, normalization, validation, utils); تبنى في `dist/` ويُعاد تصديرها في `index.ts`. |
| `packages/db/prisma/schema.prisma` | نماذج `User`, `RefreshToken`, `Analysis` (شاملة status, الأقسام الناتجة من الوكلاء، تقييمات النتيجة). |

## ٤. نقاط الحماية والملفات الحساسة
- لا تخلي هذه الملفات في الـ commit عندما تتحول الخاص:
  1. `./.env` → GEMINI/OPENROUTER API Keys، Upstash Redis URL، JWT Secret، إعدادات URL المحلية.
  2. `packages/db/.env` → Neon PostgreSQL، مفاتيح Redis، GEMINI/OpenRouter، JWT secret من بيئة التطوير.
  3. `packages/db/prisma/dev.db` → قاعدة SQLite تحتوي على بيانات المستخدمين والتحليلات.
- **باقي الملفات**: `apps/`, `packages/shared/`, `assets/`, `README.md`... هيكل المشروع وواجهة المستخدم وغيرها شيء يمكن مشاركته دون مشاكل.
- **ملاحظة هامة**: أي fork سبق ونُسخ قبل تحويل المستودع إلى خاص سيظل يرى الملفات القديمة، لذا يجب تدوير المفاتيح (JWT، Gemini، Redis، Neon) وإعلام الفريق.

## ٥. خطوات سريعة قبل التحويل إلى خاص
1. احذف `.env`, `packages/db/.env`, و`packages/db/prisma/dev.db` من Git history (filter-repo أو rebase) وأضفهم إلى `.gitignore`.
2. احتفظ بملفات `*.example` تشرح المتغيرات المطلوبة (`.env.example`, `packages/db/.env.example`).
3. قم بإصدار مفاتيح جديدة لـ Gemini، OpenRouter، Upstash Redis، Neon DB، وJWT؛ وسجّل أنها مبدلة في `SECURITY.md`.
4. تأكد أن GitHub Actions / إلخ تستخدم Secrets وليس ملفات env.
5. بعد التحويل، راجع فرق الوصول (Teams/Collaborators) وأكد أنّ CI/Deploy يُمكنه سحب Secrets دون الحاجة للـ `.env`.

## ٦. كيف تتعامل لو رجعت للملف بعد سنة
- ابدأ من القسم ١ لتتذكر أن هناك `apps/` (backend + frontend) و`packages/` (db/shared).
- راجع الجدول في القسم ٣ عندما تريد معرفة وظيفة كل ملف بسرعة.
- راجع القسم ٤ قبل أي تغيير يخص الخصوصية أو المفاتيح.
- استخدم الخطوات في القسم ٥ كـ Checklist قبل نشر النسخة الخاصة أو تعيين وصول داخلي.
