# نظرة سريعة مبسطة

## ١. لمحة عامة تخلّيك تفهم الهيكل في ثانية
- **أيقونة المشروع**: `apps/` للتطبيقات التنفيذية، `packages/` للكتب المشتركة، وملفات الجذر للضبط (pnpm، turbo، وثائق، وملفات البيئة).
- **الهدف الأساسي**: واجهة مستخدم Next.js تُرسل أفكارًا إلى Backend NestJS يتم فيها استدعاء وكلاء ذكاء اصطناعي، ثم تُعرض تقارير تحليلية شاملة.
- **أهمية الملف**: هذه الوثيقة توضح باختصار كل طبقة، تميّز الحساسيات، وتقترح خطوات قبل تحويل Repos إلى خاص.

## ٣. ملفات ومكونات رئيسية (لما تفتح الملف تعرف وش يسوي بسرعة)
| المسار | الدور المحوري |
|---|---|
| `apps/backend/src/app.module.ts` | يربط Config/Bull/Throttler والـ Modules (Auth, Analysis, Agents). يعرف الـ ThrottlerGuard الشامل. |
| `apps/backend/src/main.ts` | يفعِّل CORS، الـ ValidationPipe، وفلتر الأخطاء، ويشغّل الخادم على `process.env.BACKEND_PORT`. |
| `apps/backend/src/analysis/analysis.service.ts` | ينشئ تحليلًا جديدًا، يطبّق قيود الخطة، يستدعي وكلاء متعددين عبر الطابور ويعيد البيانات بعد انتظار التنفيذ. |
| `apps/backend/src/queue/analysis.processor.ts` | ينسق معالجة الـ job، يدير التقدم، يُشغّل الوكلاء في تسلسل، يطبّق معالجات grounding، ويحدّث سجل `analysis` النهائي في Prisma. |
| `apps/backend/src/agents/ai.service.ts` | موصل صالح لـ Gemini أو OpenRouter، يدير Redis للكاش، retries، ويستخرج JSON المستجيب من الـ Prompt. |
| `apps/backend/src/auth/*` | تسجيل/تسجيل دخول/refresh/logout باستخدام `AuthService` و`JwtService`، مع دعم Google OAuth (cookies + redirect). |
| `apps/frontend/src/app/page.tsx` | واجهة البداية، نموذج إدخال الفكرة مع قيود `MAX_IDEA_LENGTH`, يطلب تسجيل دخول قبل استدعاء `/analysis`. |
| `apps/frontend/src/app/analysis/[id]/page.tsx` | صفحة تقرير مفصل (progress polling، scoreboard، إعادة توليد الأقسام، دردشة، تنزيل PDF). |
| `apps/frontend/src/components/Sidebar.tsx` | قائمة جانبية ثابتة، أزرار Chat و Logout، وعرض بيانات المستخدم. |
| `apps/frontend/src/lib/api.ts` | axios مع Base URL من `NEXT_PUBLIC_API_URL`، Authorization header يتم جلبه من zustand/localStorage. |
| `apps/frontend/src/lib/store.ts` | zustand + persist لإدارة `token`, `refreshToken`, `user`, مع دوال `logout`, `setAuth`. |
| `packages/shared/src` | مكتبة توحيد (أنواع، تطبيع، تحقق، utils)، مبنية في `dist/` ويُعاد تصديرها. |
| `packages/db/prisma/schema.prisma` | نماذج `User`, `RefreshToken`, `Analysis` (أقسام الوكلاء، التقييمات). |

## ٤. نقاط الحماية والملفات الحساسة
- لا تخلي هذه الملفات في الـ commit عندما تتحول الخاص:
  1. `./.env` → GEMINI/OPENROUTER API Keys، Upstash Redis URL، JWT Secret، إعدادات URL.
  2. `packages/db/.env` → Neon PostgreSQL، مفاتيح Redis، GEMINI/OpenRouter، JWT secret.
  3. `packages/db/prisma/dev.db` → قاعدة SQLite محلية تحتوي على بيانات المستخدمين والتحليلات.
- أي fork موجود بالفعل يحتفظ بالنسخ القديمة؛ يجب تدوير المفاتيح (JWT، Gemini، Redis، Neon).
- بعد التحويل للخاص راجع فرق الوصول وامنح CI للأساسيات باستخدام GitHub Secrets بدلًا من ملفات env.

===== SECURITY ANALYSIS =====

### Sensitive Visibility Notes
- The overview references `.env` files (`./.env`, `packages/db/.env`) that currently store API keys (Gemini, OpenRouter), JWT secrets, Upstash Redis credentials, and Neon/PostgreSQL host data. These should never be public because they provide direct access to production/development services and authentication tokens.
- The TODO/PLAN files document backend DB mode decisions, CI expectations, and deployment tasks that may reveal internal re-architecture plans; these are ok for trusted collaborators but should stay private otherwise.
- The plan also flags the data-store ambiguity (SQLite vs PostgreSQL) and outstanding testing gaps; if exposed publicly, attackers might exploit the uncertainty during deployment or attempt to target untested endpoints.

### Recommended Visibility Handling
1. **Mark as private**, especially while sensitive `.env` contents remain in history; rotating those secrets and removing them from Git history should happen before setting the repo to public.
2. **Archive the merged content** (`.internal_master_doc.md`) in a private folder or secure vault (protected by `.private_notes/`), making sure the original four documents are deleted so the combined file is the only place to reference secrets.
3. **Keep README/SECURITY** high-level; avoid replicating API keys, but keep instructions for reporting vulnerabilities and support coverage in the public docs.

===== FINAL REPORT =====

1. **Summary**: AI Startup Analyzer is a full-stack Next.js + NestJS platform where users submit startup ideas, the backend runs multiple AI-driven agents via a BullMQ queue, stores structured analysis in Prisma, and returns scored reports for frontend consumption.
2. **Key Risks if Public**:
   - Embedded `.env` references in documentation point to valid API tokens and infrastructure credentials (Gemini/OpenRouter, Upstash Redis, Neon DB, JWT secrets).
   - DB mode ambiguity and minimal testing notes highlight gaps that could allow attackers to exploit unverified paths.
   - Ownership notes mention intact secrets in history; any fork or clone before cleaning retains direct access to services.
3. **Recommendation**: Keep this repository **Private** until secrets are rotated, `.env` contents are scrubbed from history, and documentation is rewritten to reference example variables instead of real credentials. Only then consider opening it with sanitized README/SECURITY guidance.
